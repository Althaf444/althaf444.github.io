1. Dashboard Module

This module acts as the primary router for the application, handling user navigation and dispatching API requests to three main backend services.

Actors/Layers: Client (Frontend UI), Backend (divided into three sub-services: Transaction, Budget Limits, Analytics).

Key Workflows:

Workflow A: Add Transaction:

Frontend: User selects "Add transaction" → inputs amount/date → toggles Income/Expense → hits "Save".

Backend (API Request): Receives the New Transaction Payload.

Logic: Validates input. If valid, saves to the database → recalculates total balance → checks against the budget category limit. If the limit is exceeded, it generates an "over budget" warning message.

Response: Returns a success message (and optional warning) to update the UI.

Workflow B: Budget Management:

Frontend: User navigates to the "Budgets" tab → adjusts category limits → saves.

Backend (API Request): Receives the New Budget Configuration.

Logic: Validates input → updates configuration in the database.

Response: Returns a success message to the frontend.

Workflow C: View Analytics:

Frontend: User navigates to the "Reports" tab → selects a time period → views charts.

Backend (API Request): Receives the request for statistics (date range).

Logic: Fetches transactions for the selected dates → calculates total spending per category.

Response: Returns aggregated data arrays to populate the frontend charts.

2. Login & Authentication Module

This module dictates the user onboarding and session management flows.

Actors/Layers: User/Guest (Frontend), System (Backend/Auth Service).

Key Workflows:

Workflow A: Existing User Login:

User inputs credentials → System authenticates → User is logged in.

Workflow B: New User Registration:

User selects "Create account" → fills in registration details (email, password, etc.).

Backend: Creates a new account record (likely flagged as inactive/unverified) → dispatches a Verification Email.

Workflow C: Email Verification Loop:

The system waits for email confirmation.

If the user fails to verify, they can trigger a "Retry" to resend the email.

Once verified via a confirmation link, the system completes the registration, activates the account, and routes the user to a successful login state.

3. Budget Management Module

This module handles the CRUD operations for user budgets and the internal logic for tracking threshold violations.

Actors/Layers: User (Frontend), Budgeting System (Backend).

Key Workflows:

Data Entry & Validation: User opens the dashboard, creates or edits a budget, and submits details. The backend immediately validates the input. If invalid, it returns an error state to the UI.

Core Processing Loop:

If valid, the backend updates the budget records.

It then triggers an internal function to "Track Expenses against Budget" and "Calculate Budget Usage" (this requires fetching sum totals of categorized expenses).

Conditional Branching (Limits):

Condition A (Exceeded): If usage > limit, the system prepares a warning, displays the budget summary, and flags the UI.

Condition B (Safe): If usage ≤ limit, it triggers a standard notification (e.g., "Budget updated") and displays the summary.

4. Transaction Module

This diagram expands on the transaction flow briefly seen in the dashboard, detailing the exact data exchange.

Actors/Layers: User (Frontend), System (Backend).

Key Workflows:

Frontend Input: User enters details, flags as Income/Expense, and submits.

Backend Validation & Error Handling: Receives payload. If invalid (e.g., missing fields, negative numbers where not allowed), it immediately aborts and returns an error message to the UI.

Database Transaction: If valid, the transaction is committed to the database.

Post-Save Triggers:

Recalculate the user's total balance.

Check the specific category limit. If exceeded, append an "over budget message" to the payload.

Response: The API responds with success, and the UI closes the transaction modal/app state.

5. Bank Connect & Sync Module

This is the most complex diagram, representing a 3-legged OAuth flow typical of open banking integrations (like Plaid or Tink).

Actors/Layers: User, Application (UI & Backend), Data Aggregator (Third-Party API).

Key Workflows:

Phase 1: Initialization: User clicks "Add bank". The App Backend requests a "Link Token" from the Data Aggregator. The token is sent to the App UI to initialize the banking widget.

Phase 2: User Authentication: The UI displays the bank list. The user selects a bank, enters credentials, and handles MFA (Multi-Factor Authentication) if required. Crucially, these credentials go directly to the Data Aggregator via the secure widget, not your backend.

Phase 3: Token Exchange: Once the user grants consent, the Data Aggregator validates with the financial institution and passes a Public Token back to your App UI.

Phase 4: Securing Access: Your App UI forwards this Public Token to your App Backend. Your backend securely requests to exchange this public token for a permanent Access Token from the Data Aggregator.

Phase 5: Syncing: Your backend persists the connection (saves the Access Token to the user's profile in the DB) and immediately fires off a request to fetch the initial financial data (transactions, balances). The UI displays a success message.
